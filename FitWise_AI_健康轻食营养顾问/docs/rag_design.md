# RAG 检索增强生成 — 设计文档

## 一、架构概览

```
          ┌──────────────────────────┐  ┌──────────────────────────┐
          │     Milvus 向量数据库      │  │    Elasticsearch BM25     │
          │  COSINE + IVF_FLAT        │  │   倒排索引 / standard分词  │
          │  embedding (1024-dim)     │  │   content / source / ...  │
          └──────────┬───────────────┘  └──────────┬───────────────┘
                     │ top10 向量相似度             │ top10 BM25 关键词
                     └──────────┬──────────────────┘
                                │ 去重合并 → 加权融合 65/35
┌──────────┐    ┌───────────┐    ┌───────────┐    ┌──────────┐    ┌──────────┐
│  load()  │ ──▶│ retrieve  │ ──▶│  rerank() │ ──▶│  LLM生成 │ ──▶│  返回    │
│  预处理   │    │  多路召回  │    │  精排 3    │    │ answer() │    │ 答案+来源 │
└──────────┘    └───────────┘    └───────────┘    └──────────┘    └──────────┘
      │               │                │                 │
  ┌───┴───┐      ┌───┴───┐       ┌───┴──────┐     ┌───┴────┐
  │文本切分│      │ES BM25│       │Cross-Encode │     │Prompt  │
  │Embedding│     │Milvus │       │Reranker  │     │OpenAI  │
  │双写ES+MV│     │→融合  │       │→相关性分数 │     │降级兜底│
  └───────┘      └───────┘       └──────────┘     └────────┘
```

## 二、为什么这么设计

### 2.1 文档加载 (`load`)

| 决策                                     | 理由                                                                                                                         |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| **启动时一次性加载到内存**               | 知识库规模小（百级文档），无需外部向量数据库，减少运维依赖                                                                   |
| **`chunk_size=520` 字符 + `overlap=90`** | 经验值，520字符约对应 200~500 token，6个chunk拼在一起控制在 LLM 上下文窗口的合理范围内；90字符重叠避免关键信息被切在两块边界 |
| **CSV 一行一个 chunk**                   | 食物营养表每行是一个独立实体（鸡胸肉/鸡蛋），再切会丢失结构完整性                                                            |
| **`infer_domain` 文件名关键词分类**      | 简单直接，适配现有文件命名规范，无需额外标注                                                                                 |
| **`.md/.txt` 和 `.csv` 两种处理**        | 知识库是自然语言文章，食物库是结构化表格，分开对待更合理                                                                     |

### 2.2 语义向量化 (`_batch_embed`)

| 决策                                             | 理由                                                                                    |
| ------------------------------------------------ | --------------------------------------------------------------------------------------- |
| **`text-embedding-v3` 替代 `HashingVectorizer`** | 哈希向量只是字符 ngram 碰撞，无语义；Embedding API 真正理解"鸡胸肉"和"鸡里脊"的语义关系 |
| **批量 API 调用，batch_size=20**                 | 每次请求最多 20 条文本，避免单次请求过大，也避免过多网络往返                            |
| **失败槽位填零向量**                             | 部分文本嵌入失败不阻塞整体流程，零向量在余弦相似度中得分为 0，自然排在最后              |
| **启动时一次性生成并缓存**                       | `load()` 中批量生成所有文档的 embedding 存到 `self.embeddings`，检索时不再调 API        |
| **无 API Key 降级全零向量**                      | 此时检索退化为纯 BM25，仍有基本可用性                                                   |

### 2.3 Milvus 向量存储 (`_init_milvus` / `_vector_scores_milvus`)

Embedding 从内存 numpy 矩阵迁移到 Milvus 分布式向量数据库，一次构建持久化，重启不重新计算。

| 决策 | 理由 |
|------|------|
| **Milvus COSINE + IVF_FLAT** | IVF_FLAT 适合十万级以下文档，COSINE 与 Embedding API 的度量一致 |
| **`load()` 增量判断** | 对比 Collection 中已有实体数 vs 扫描到的文档数，相等则跳过重建 |
| **`_batch_embed` 仍保留** | 查询向量生成不变；构建时批量调用 Embedding API 写入 Milvus |
| **`_vector_scores_numpy` 降级** | Milvus 连接/搜索异常 → 自动回退内存 numpy 全量计算，零中断 |
| **`pymilvus` 可选导入** | 没装 pymilvus 时服务正常启动，`_milvus_ok=False`，走 numpy 降级 |
| **元数据随向量存储** | content / source / domain / chunk_id 全部存入 Milvus，检索时一并返回 |

### 2.4 BM25 关键词检索

| 决策                                             | 理由                                                                                                         |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------ |
| **向量为主 (65%) + BM25 (35%)**                  | 向量擅长语义同义匹配（"瘦身"≈"减脂"），BM25 擅长精确术语命中（"BMI=28"），互补                               |
| **BM25 参数 `k1=1.5, b=0.75`**                   | 标准配置，b=0.75 对长文档做中等长度惩罚                                                                      |
| **IDF 用平滑公式 `ln(1 + (N-df+0.5)/(df+0.5))`** | BM25 经典变体，`+0.5` 避免除零，`1+` 保证 IDF ≥ 0                                                            |
| **`_build_bm25` 预计算**                         | 把 `doc_tokens`、`idf`、`avgdl` 提前算好缓存，检索时只做 O(N_query_tokens × N_docs) 的打分循环，不重复建索引 |

### 2.4 检索 (`retrieve`)

| 决策                       | 理由                                                                        |
| -------------------------- | --------------------------------------------------------------------------- |
| **max 归一化而非 softmax** | 场景关心相对排名而非概率分布，max 归一化简单高效，不引入 softmax 的计算开销 |
| **默认 `top_k=6`**         | 6 个 chunk 的证据量够 LLM 做判断，又不会撑爆上下文窗口                      |
| **`temperature=0.2`**      | 知识问答场景需要事实准确性，低温度减少 LLM 幻觉                             |

### 2.5 Rerank 精排 (`rerank`)

粗排召回 10 条候选，精排用专用 Rerank 模型做交叉编码（Cross-Encoder）重排序：

```
粗排 10 条候选
    │
    ▼
┌─────────────────────────────────┐
│  POST /v1/rerank                │
│  {                              │
│    "model": "qwen3-rerank",     │  模型：env NUTRIFIT_RERANK_MODEL
│    "query": "用户问题",          │
│    "documents": ["doc1", ...]   │  每个 document 截断到 800 字符
│  }                              │
│  → { results: [                 │
│      {index: 2, relevance_score: 0.92},  ← 问题+文档2 相关性最高
│      {index: 5, relevance_score: 0.87},
│      {index: 0, relevance_score: 0.74},
│    ]}                            │
└─────────────────────────────────┘
    │
    ▼
按 relevance_score 降序 → 取前 3 → 送入 answer()
```

| 决策                   | 理由                                                                                 |
| ---------------------- | ------------------------------------------------------------------------------------ |
| **专用 Rerank 模型**   | Cross-Encoder 同时看 query+doc，比 Bi-Encoder（向量）和 BM25（关键词）更懂语义相关性 |
| **query+doc 两两配对** | 每个候选独立打分，不受其他候选干扰，分数可比较                                       |
| **文档截断 800 字符**  | 控制请求 body 大小，800 字符覆盖 chunk 主要内容                                      |
| **粗排 10 → 精排 3**   | 粗排保证召回率，精排保证准确率，比直接粗排 6 条质量高                                |
| **三重降级**           | 无 API Key → 粗排截断；API 异常 → 粗排截断；部分结果缺失 → 补 0 分                   |

### 2.6 查询改写 (`_rewrite_query`)

在检索入口前对用户问题做一次轻量改写，把口语化问题转成关键词明确、适合检索的短查询。

```
用户问：我最近想瘦一点，但是不知道每天吃多少合适
    │
    ▼  _rewrite_query()
改写后：减脂人群每日建议摄入量是多少？
    │
    ▼  retrieve(改写后) → rerank(改写后) → answer(原始问题)  ← 回答仍用原文
```

| 决策                                | 理由                                                           |
| ----------------------------------- | -------------------------------------------------------------- |
| **检索用改写后，回答用原文**        | 改写提升召回质量，但回答时需要原始意图和语气，两者各司其职     |
| **低 temperature + max_tokens=120** | 改写需要稳定、简短，不鼓励 LLM 发挥                            |
| **Prompt 简单直接**                 | 最小化改写本身的 token 消耗                                    |
| **四重降级**                        | 无 API Key → 原句；LLM 超时 → 原句；返回空 → 原句；异常 → 原句 |

### 2.7 事实溯源校验 (`_fact_check`)

生成回答后，逐句检查是否有证据支持，删除无依据的句子，防止 LLM 幻觉。

```
LLM 生成原回答
    │
    ▼  _split_sentences()
["减脂期建议每日摄入 1500kcal。", "鸡胸肉是优质蛋白来源。", "建议多喝水。"]
    │
    ▼  每句 × 证据（sources[:6]）→ LLM 逐一判定
[
  {index: 1, verdict: "可信"},    ← 证据中有摄入量数据
  {index: 2, verdict: "可信"},    ← 证据中有鸡胸肉营养信息
  {index: 3, verdict: "可疑"},    ← 证据中未涉及饮水相关内容
]
    │
    ▼  删除可疑句子，拼接可信句子
"减脂期建议每日摄入 1500kcal。鸡胸肉是优质蛋白来源。"
```

| 决策 | 理由 |
|------|------|
| **逐句校验** | 粒度细，只删问题句不伤及整段回答 |
| **风险提示/免责声明豁免** | 安全兜底句子即使无证据也保留，不误删 |
| **LLM 异常 → 返回原回答** | 校验失败不阻塞主流程，checked=false 标记 |
| **无 API Key → 跳过校验** | 降级模式本身就无 LLM，不需要校验 |

### 2.8 无 API Key 降级

当 `OPENAI_API_KEY` 未配置时，直接从检索结果截取前 160 字符作抽取式摘要。保证离线环境也能看到「知识库里有什么」，不至于完全不可用。

## 三、局限

### 3.1 检索质量

| 局限                      | 影响                                                     |
| ------------------------- | -------------------------------------------------------- |
| **精排额外开销**          | Rerank API 每次多一次网络调用，延迟 +0.5~1s              |
| **改写增加延迟**          | 查询改写多一次 LLM 调用，+0.5~1s，有降级兜底不影响可用性 |
| **无负反馈 / 相关性标注** | 没有数据闭环，无法根据用户反馈优化排名                   |

### 3.2 工程能力

| 局限                     | 影响                                                               |
| ------------------------ | ------------------------------------------------------------------ |
| **BM25 仍在内存中全量计算** | BM25 无倒排索引，万级以上文档时延迟不可接受 |
| **无增量更新**           | 新增/修改知识库文件需要重启服务才能生效 |
| **单例全局**             | `rag_service = RAGService()` 模块级单例，无法按租户/场景隔离知识库 |
| **Milvus 为可选依赖**    | 未安装/未启动时退化为内存 numpy，高并发下内存和 CPU 压力大 |

### 3.3 Prompt / 生成

| 局限            | 影响                                                           |
| --------------- | -------------------------------------------------------------- |
| **Prompt 简单** | 没有 few-shot 示例，没有对模糊问题的澄清机制                   |
| **无对话历史**  | 每次 `answer()` 是独立调用，不关联前文，多轮对话下可能重复询问 |

## 四、升级路线

### 第一阶段：快速优化（低成本，高收益）

- [x] **接入中文分词库**：用 `jieba` 替换 `tokenize` 的正则分词，让 BM25 匹配到完整词组 ✅ 已完成
- [x] **粗排 top_10 → Rerank 精排 top_3**：Cross-Encoder Reranker（`qwen3-rerank`），query+doc 两两配对打分，按 relevance_score 降序 ✅ 已完成
- [x] **替换为 LLM Embedding**：用 `text-embedding-v3` 替代 `HashingVectorizer`，语义向量显著提升召回质量 ✅ 已完成
- [x] **添加查询改写**：`_rewrite_query` 检索前用 LLM 把口语化问题改写为检索友好短查询，降级回原始问题 ✅ 已完成
- [x] **引用校验**：`_fact_check` 生成后逐句溯源校验，可疑句子标记并删除 ✅ 已完成

### 第二阶段：架构升级（中期）

- [x] **迁移到向量数据库**：Milvus（COSINE + IVF_FLAT），持久化向量存储，支持增量写入、重启不重建 ✅ 已完成
- [x] **混合多路召回**：ES BM25 top10 + Milvus 向量 top10 → 去重融合 → Rerank 精排，四级降级链 ✅ 已完成
- [ ] **HyDE（假设文档嵌入）**：先让 LLM 为问题生成一个假设答案，用假设答案的 embedding 去检索，对抽象问题效果显著
- [ ] **知识库热更新**：文件监听 / API 触发 reindex，不重启服务

### 第三阶段：完整 RAG 系统（长期）

- [ ] **多轮 RAG**：关联对话历史，支持追问（"那三文鱼呢？" → 上下文推断在问热量）
- [ ] **意图路由**：不同问题走不同检索策略（营养查询 → 食物库，减脂建议 → 知识库，食谱生成 → 食谱库）
- [ ] **用户反馈闭环**：记录用户对回答的采纳/纠正，用对比学习微调检索模型
- [ ] **知识图谱融合**：从食物数据库构建实体关系（食物→营养素→健康影响），支持结构化推理
